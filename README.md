# 逆天 Agent Hub · AI Trading Demo

> 一个跑在 **Bitget UTA V3 模拟盘** 上的对话式交易 Agent —— 用一句自然语言生成策略、扫描行情、下单、自动 tick，全程在聊天里完成。
>
> Demo: <https://frontend-666788.vercel.app/>

---

## 第一段 · 思路

### 为什么做这个

主流交易工具有两个让我难受的地方：

1. **策略写起来太重** —— 想验证一个"WLD 突破 20 日均线且成交量放大 1.5 倍时买入 40% 仓位"这种很朴素的想法，要么写 Pine、要么写 Python、要么去配可视化策略，门槛远高于"我脑子里那句话"。
2. **风控/退出和"开仓灵感"是脱钩的** —— 大部分 AI 助手只负责"给你看个信号"，真正的止损、追踪止盈、仓位再平衡仍然要人手操心，结果就是策略没死在市场上，先死在操作疲劳上。

所以我做了 **逆天 Agent**：把一段自然语言策略变成一个**活的 Agent**——它有自己的感知节奏、自己的决策节奏，自己负责退出。用户只需要写策略 + 看面板。

### 策略为什么成立 / 感知了哪些信号

Agent 采用 **"小信号集 + 严格风控"** 这条相对保守的路线，而不是堆几十个指标：

| 维度 | 信号 | 用途 |
|------|------|------|
| 技术面 | EMA20 / EMA60、布林带、ATR、20D Volume MA | 趋势 / 波动率 / 异常放量 |
| 行情结构 | Bitget Spot + Perp 的最近 N 根 K（默认 1h / 4h） | 突破识别、网格区间锚定 |
| 持仓上下文 | UTA 账户的余额、持仓、未完成委托 | 真实可下单仓位、避免重复开仓 |
| 用户意图 | 对话里抽取的标的 / 触发条件 / 仓位 / 网格参数 | 生成策略 JSON（DSL） |

我**故意没接情绪流和链上数据**，原因是这个 Demo 想验证的是"对话 → 策略 → 执行 → 风控"这一整条链路是否顺滑；信号越多，调试越发散，反而不利于看清问题。后续会按 *Roadmap* 接入。

### 决策逻辑

Agent 内部把一次"心跳（tick）"拆成三层职责，三层都是独立可替换的子节点：

```
            ┌─────────────────────────────────────────────┐
 自然语言   │  1. 策略生成器（LLM, JSON 输出）             │
   ───►     │     - 抽取 symbol / 触发 / 仓位 / 失效条件    │
            │     - 落到策略 DSL                            │
            └────────────────────┬────────────────────────┘
                                 │
            ┌────────────────────▼────────────────────────┐
 行情/账户  │  2. 感知（Perception, 每 tick 刷新）          │
            │     - 拉 K线 + 指标 + 账户上下文              │
            │     - 输出 "is_trigger / score / context"     │
            └────────────────────┬────────────────────────┘
                                 │
            ┌────────────────────▼────────────────────────┐
 风控规则   │  3. 决策·风控·退出（Decision, tick 调用）     │
            │     - 仓位检查、单笔上限、最大回撤            │
            │     - 通过 Bitget MCP / API 下单              │
            │     - 维护止损 / 追踪止盈 / 网格再平衡        │
            └─────────────────────────────────────────────┘
```

UI 里能看到 Agent 卡片上挂着两个状态标签 —— **"感知 · 待拉取"** 和 **"决策·风控·退出 · tick 时调用"** ——就是上面 2、3 两层的运行时状态。

### 风控如何设计

风控**默认开启**、不可被 LLM 单独关闭，由执行层强制校验：

- **单标的最大仓位** ≤ 用户写的目标仓位（例："买入 40% 仓位" → 触发上限 40%，不会因 LLM 失误连续加仓）
- **单笔最大下单额** 由账户净值 × 阈值动态计算
- **硬止损** = 入场价 ± k×ATR（默认 k=2），止损单在开仓后立刻挂出，不等 tick
- **网格策略** 必须给出 *上下边界 + 单格仓位*，未给出时拒绝执行
- **失败回滚** —— 下单 API 报错时整笔交易作废，Agent 不重试，等下一个 tick 重新评估
- **模式锁定** —— 全程跑在 Bitget UTA V3 模拟盘（Paper 模式），UI 顶部常驻 "Paper 模式" 徽章，杜绝误连实盘

### 交易标的

合约 + 现货均可，目前在 Demo 里用得最多的是 **BTC / WLD / PEPE / AAVE**：

- BTC：趋势 + 突破策略（验证大盘流动性下的执行）
- WLD / AAVE：突破 + 量能确认（验证中市值偏波动的标的）
- PEPE：网格策略（验证高波动、均值回归型标的）

- ✅ **执行层** —— 描述策略 / 扫描行情 / 买卖下单 / 自动 tick **全部在对话里完成**
- ✅ **三层 Agent 节点** —— 策略生成 / 感知 / 决策·风控·退出，状态在 UI 上实时显示
- ✅ **风控硬约束** —— 仓位上限、ATR 止损、网格边界校验、Paper 模式锁定
- ✅ **多策略示例** —— BTC 趋势、WLD 突破放量、PEPE 网格、AAVE 切换
- ✅ **账户面板** —— 连接状态、Key 掩码显示、切换 / 退出账户

### 尚未实现

- ⬜ **多 Agent 协作** —— 现在是单 Agent 跑单策略，计划接 Agent Hub 做"策略池 + 调度器"
- ⬜ **回测引擎** —— 目前只有 forward test，回测报告还没补
- ⬜ **链上 / 情绪信号** —— 留好了接口位（perception 是插件式的），但还没接 DeFi TVL、X 情绪、链上巨鲸流向
- ⬜ **策略持久化** —— 当前重启后策略要重发
- ⬜ **告警通道** —— Telegram / Webhook 推送

### 开发中遇到的问题与解决

| 问题 | 解决 |
|------|------|
| LLM 生成的策略 JSON 偶尔丢字段（例如忘了写仓位上限） | 在策略 DSL 上加 schema 校验，缺字段直接退回让 LLM 补 —— **不靠 prompt 哄它，靠校验逼它** |
| Bitget UTA V3 模拟盘下单参数和文档有几处对不上（合约方向 / 保证金模式） | 写了一层薄薄的 `bitget-core` 适配，统一 normalize 输入；所有错误返回收敛成同一种 Error 让 Agent 重新决策 |
| Tick 频率太快会重复开仓 | 在风控层加 *idempotency key*（symbol + 策略 id + tick 编号），同一 tick 内重复决策只下一单 |
| 凭证安全 | 凭证只放后端进程内存 + `.env`，前端只看到掩码（如 `bg_1fc…63a0`），UI 顶部明示"Paper 模式" |

### 用到的框架 / 模型 / API / Bitget 工具

- **前端**：React + Vite，部署在 Vercel
- **后端**：Node.js，自研 `bitget-core` 适配层
- **交易所**：Bitget UTA V3（`api.bitget.com`）—— **模拟盘 / Paper**
- **LLM**：用于策略生成与对话理解（JSON-mode 强约束输出）
- **指标**：内置 TA 引擎（EMA / 布林 / ATR / Volume MA / 突破检测）

**Bitget 工具使用情况**：

- 🟦 **Agent Hub** —— 用作 Agent 的运行容器与状态展示（UI 上的 "逆天 Agent" 卡片即对应 Hub 里的 Agent 实例）
- 🟦 **MCP Server** —— 通过 MCP 调用 Bitget 行情 / 账户 / 下单接口，作为执行层的统一通道
- 🟦 **Skill Hub** —— 把"扫描行情"、"生成策略"、"下单"、"tick 评估"封装成可复用 Skill，挂到 Agent 上

（暂未使用 Playbook，下一步会把"开仓 → 加仓 → 退出"的常见组合做成 Playbook 模板。）

## 第三段 · 对 AI Trading 的看法

### Bitget AI 工具的体验

- **Agent Hub** 把 Agent 的"运行时 + UI"统一了，少写很多胶水，做 Demo 非常顺
- **MCP Server** 是这次最舒服的部分 —— 它让"调用交易所"和"调用本地工具"长得一样，Agent 不需要为不同后端写不同的 client，可组合性强
- **Skill Hub** 让 Agent 的能力变成"可组合积木"，迭代时只动 Skill 不动 Agent，这点在快速试错阶段非常有价值

### 改进建议

1. **MCP Server 的错误语义**希望更统一：目前不同 endpoint 的错误结构不完全一致，Agent 要写一些 "if 这个字符串 then 重试" 的脆弱逻辑，期待标准化的 *error code + retryable flag*
2. **Playbook 与 Agent Hub 的状态可见性**：希望 Playbook 步骤可以以 trace 的形式直接在 Hub 里看到当前执行到哪一步、卡在哪里
3. **模拟盘的 fill 行为**与实盘更贴近一些：希望加入滑点 / 部分成交的模拟选项，这样回测 → 模拟 → 实盘的过渡能更平滑

### 对 Agentic Trading 的判断

我的判断是 Agentic Trading 短期不会取代量化团队，但会**抹平"想法 → 上线"的成本**，这件事的意义比"自动赚钱"更大：

- **资产端**：个人/小团队第一次有能力把"一个朴素的市场观察"快速变成一个**有风控的、能 24h 跑的策略**，而不是停留在群聊里。
- **平台端**：交易所从"撮合 + API"升级为"撮合 + API + Agent 运行时"，会出现"策略市场"这种新形态 —— 谁的 Agent 拿稳定 Sharpe，谁拿订阅费。
- **风险端**：LLM 仍然会**自信地写错策略**。所以未来一年最重要的不是"更强的模型"，而是 **"更强的策略 DSL + 更硬的风控层"** —— 把 LLM 关在笼子里干活，而不是放出去自由发挥。

简而言之：**模型负责想象力，DSL 和风控负责活下去。** 逆天 Agent 就是按这条路线设计的。
